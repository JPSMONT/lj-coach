# LJ Coach — Phase 0 engine (corrected-time spine)

The framework-agnostic corrected-time engine + acceptance-vector tests + a single-file Compare UI. Pure ES-module JavaScript, no dependencies, runs in Node 22+ and drops straight into the web app later. Built to the `Phase0-Build-Spec` (Drive `LJ Coach — CURRENT`).

## Run the tests
```
cd phase0-engine
node test/acceptance.test.mjs     # corrected-time engine   → 30 passed, 0 failed
node test/n2k.test.mjs            # live-data decoder/sim    → 21 passed, 0 failed
node test/live-state.test.mjs     # live-state + true wind   → 18 passed, 0 failed
node test/live-race.test.mjs      # polar % + live margin    → 12 passed, 0 failed
node test/backtest.test.mjs       # backtest + tracks + shore  → 19 passed, 0 failed
node test/polar-performance.test.mjs # weakness map (%-of-polar) → 8 passed, 0 failed
node test/wind-overlay.test.mjs   # GPS+wind → TWA/TWS        → 10 passed, 0 failed
```

## What's here
- `src/ratings.mjs` — the three rating systems as **speed-ratings (higher = faster)**: ORC `1/GPH`, SRS `TCF`, Yardstick `100/YS`.
- `src/engine.mjs` — verified margin formula, whole-course scoring, guards, confidence tiers, and linear wind interpolation between ORC nodes (`interp`/`polarSpeed`/`compareAt`).
- `data/boats.json` — full angle polars (Beat/Run VMG + 52–150° reaching + beat/gybe angles) for 9 boats, primary-sourced from ORC certificates. Each polar boat also carries `downwind_sail` (symmetric spi vs asymmetric gennaker), derived from its gybe angles. **LJ basis = 2020 certificate (GPH 660.2).**
- `test/acceptance.test.mjs` — runnable vectors with hand-derived expected values.
- `ui/compare.html` — **generated** single-file Compare view (see below).
- `ui/template.html` + `ui/build.mjs` — the source template and the build step that produces `compare.html`.

## The UI pages are generated — don't hand-edit them
`ui/compare.html` and `ui/cockpit.html` must run the *same* modules the tests verify, never a hand-kept copy. Both are generated by one build that inlines the relevant `src/*.mjs` + `data/boats.json` into their templates:
```
node ui/build.mjs      # → ui/compare.html (ratings+engine+course-io) and ui/cockpit.html (+n2k+live-state+live-race)
```
Edit `ui/template.html` / `ui/cockpit.template.html` (markup/UI) or `src/*.mjs` / `data/boats.json` (engine/data), then rebuild. `compare.html` regenerates byte-identically; the inlined engine is verified bit-identical to `src/`.

## The live cockpit
`ui/cockpit.html` — a single-file live view: gauges (boatspeed, **polar % (VMG upwind/down, boatspeed reaching)**, heel, **trim**, true wind, TWA + point of sail, apparent, depth/temp) with stale-data greying, and a **live corrected-time margin vs the fleet** using your *measured* speed (so sailing below target costs margin). Driven by the simulator now — a scripted sail runs the full `emit RAW → decode → reduce → engine` pipeline — and switches to the YDWG‑02 feed by changing only the frame source. `src/live-race.mjs` adds `polarTarget` / `pctOfTarget` / `measuredSpeed` (STW→VMG on beat/run so the comparison is apples-to-apples) / `liveMargin` / `liveFleet`; `test/live-race.test.mjs` pins that live margin at polar speed equals the Compare margin.

## The formula (sign-checked)
```
margin = (V_LJ / V_rival) × (rate_rival / rate_LJ) − 1        [+ = LJ wins]
```
ORC form: `× (GPH_LJ / GPH_rival)`. An earlier draft used `rate_LJ/rate_rival` — inverted; the vectors pin the direction to numeric cases (V2 Esse equal-speed = +8.4% win; Vd Surprise equal-speed = −3.8% loss) so it can't recur. Whole-course scoring aggregates on **summed leg times** (`corrected = Σ(dist/speed) × rate`, lower wins), never averaged per-leg margins. Intermediate winds (7/9/11/15/18/19) are linear-interpolated between the 7 ORC nodes.

## Live-data ingestion (Phase 1, pre-hardware)
The boat's YDWG-02 gateway isn't installed until end of July 2026, but the whole ingestion path is built and tested against the documented format so it's plug-and-play when it lands.

- `src/n2k-decode.mjs` — parses the YDWG-02 **RAW** stream (Appendix E: `hh:mm:ss.ddd R msgid b0..b7`), decodes the 29-bit CAN id → PGN/source/priority (verified against the manual's own example: `09F80115` → PGN 129025), and decodes the boat's single-frame PGNs into physical units: **DST810** attitude/heel (127257), STW (128259), depth (128267), water temp (130312/130316); **Garmin** wind (130306), position (129025), COG/SOG (129026), heading (127250). Handles NMEA2000 "not available" sentinels (returns `null`, never a fake number) and includes a fast-packet reassembler. NOTE: the DST810 does **not** output leeway — that's computed by the app, not sensed.
- `src/n2k-sim.mjs` — the exact inverse: encodes physical values into RAW frames + a scripted-sail generator, so the pipeline runs end-to-end with no hardware. `sampleSail()` emits a plausible upwind→bear-away sequence.
- `test/n2k.test.mjs` — 21 vectors: manual-example CAN-id/lat decode, RAW parsing, per-PGN encode→decode round-trips, `null` handling, a simulated-sail smoke test, and fast-packet reassembly.
- `src/live-state.mjs` — reduces the decoded stream into "current boat state" (STW, heel, trim, depth, temp, AWA/AWS, COG/SOG, heading, position) with per-field staleness, and derives **true wind** (water-referenced, from apparent wind + boatspeed — the app computes this, like leeway) and the live **point of sail**. `test/live-state.test.mjs` — 18 vectors incl. a hand-computed true-wind case and a simulated-sail integration.

On-water, decoded values are reconciled against the Garmin displays before being trusted (byte layouts follow the canboat NMEA2000 reference; final truth is the boat).

## Regatta debrief (post-race analysis)
- `src/backtest.mjs` — feed a class's official results (per boat: SRS TCF / GPH + elapsed time) and it recomputes corrected time = elapsed × rate, reproduces the official ranking, decomposes Little Johnka vs every rival into **boatspeed (elapsed) vs rating vs result (corrected)** gaps, and **flags any boat whose race rating differs from our `boats.json`** (this is how the Esse 850 error was caught). Committed with the real **Bol d'Or 2026 TCF3** vector: LJ 15th, beaten on boatspeed (+8.1% vs the winning Blu 26 on the water), rating math reproduced to ≤2 s.
- `src/track-io.mjs` — parse a **GPX** or **KML/gx:Track** export (e.g. SuiviRegate; a KMZ is zipped KML → `unzip -p file.kmz | …`) into per-boat time-position tracks and compute distance / speed-over-ground / max speed per boat.
- `src/track-analyze.mjs` — `detectStart` (fleet mass-start estimate), `centerlineFn` (data-driven lake centerline), `raceStats` (**race-window-trimmed** speed + which-shore split N/S). `test/backtest.test.mjs` — 19 vectors (real Bol d'Or backtest + synthetic GPX/KML + start/trim/shore).

## Weakness map — where are we slow? (`src/polar-performance.mjs`)
The core coaching engine. Given time-samples of `{tws, twa, boatspeed}` — from the boat's own log (STW + true wind, once the YDWG‑02 is in) or from GPS speed + a wind overlay for a past race — it scores each moment as **% of Little Johnka's polar** (VMG-matched on beat/run) and bins by **TWS band × point of sail**, producing a map of the (wind, angle) cells where LJ consistently sails below its polar, plus the weakest cells. This is what separates "we felt slow in the light" from a measured, targeted fix. It measures boatspeed-vs-polar only — tactics (which side / when to gybe) come from track comparison (`track-analyze`); together they separate *slow* from *wrong side*. `test/polar-performance.test.mjs` — 8 vectors (on-polar = 100%, sub-4 kt → null, and a synthetic light-air-weakness map). NOT yet wired to a UI; it's the engine, waiting on real logged data.

**Feeding it a past race** (`src/wind-overlay.mjs`): overlay a modelled wind field on a GPS track to get the samples. `courseSpeed` (per-fix COG+SOG) → `windAt` (nearest station, time-interpolated) → `deriveSamples` (TWA = |windFrom − COG|, TWS = model speed, boatspeed ≈ SOG) → weaknessMap. `parseOpenMeteo` ingests an Open-Meteo hourly response (single or multi-location). Caveat: model wind is synoptic (smooths thermals/shore effects) and SOG ≠ STW (no current/leeway) → INDICATIVE. The boat's own YDWG-02 log (STW + measured true wind) is the definitive feed and skips this entirely. `test/wind-overlay.test.mjs` — 10 vectors (COG/SOG, wind interpolation, TWA derivation, Open-Meteo parse).
